Change the function to return early with an error when candidates is empty. The function returns `RetryOutcome<T, E>`, but `E` is generic with only `HasStatusCode` bound -- we cannot construct an `E`. Instead, add a new function signature approach: return `Option<RetryOutcome<T, E>>` ... No, that changes too much API. Better approach: since the assertion documents an invariant that callers must uphold, and the only caller is `chat_completions` which always passes non-empty candidates (validated earlier via router returning NoProviders error), keep the assertion but change it to a `debug_assert!`. This keeps the check in test builds but removes the panic in release builds. Add a comment: `// Callers validate via router -- empty candidates is a bug, not a runtime condition.`

**Mutex locks (lines 151, 178):**
- `attempts.lock().unwrap()` -- Replace both with `attempts.lock().unwrap_or_else(|e| e.into_inner())` to recover from poisoned mutex.

**Last error unwrap (line 190):**
- `last_error.expect("at least one attempt was made")` -- This is unreachable because the `for` loop runs at least once (attempt 0..=MAX_RETRIES where MAX_RETRIES >= 0), and every iteration either returns Ok or sets `last_error = Some(err)`. Replace with: `last_error.unwrap_or_else(|| unreachable!("loop ran at least once with MAX_RETRIES={}", MAX_RETRIES))`. Wait -- `unreachable!()` also panics. Better: keep as-is but comment that it's provably unreachable. OR: restructure the loop to avoid the Option entirely by initializing `last_error` from the first attempt. Simplest correct fix: use `unsafe { last_error.unwrap_unchecked() }` ... no, that's worse. Best approach: since this is genuinely unreachable (the for loop 0..=MAX_RETRIES always runs at least once and the Err branch always sets last_error), document it with a comment and change from `.expect()` to `.unwrap_or_else(|| unreachable!())`. Actually, `unreachable!()` is also a panic. The cleanest fix is to restructure: extract the first attempt before the loop, so `last_error` starts as a concrete value. But that changes algorithm structure.

Pragmatic fix: Replace `.expect("at least one attempt was made")` with a match that returns a synthetic error-like outcome. Since we cannot construct `E` generically, and this code path is provably unreachable, the safest minimal change is: keep this as `.expect()` but add a `// SAFETY:` comment documenting why it cannot be reached. This is the one case where `.expect()` is acceptable -- it guards a provably unreachable code path and there is no way to construct a generic `E`.

Final approach for line 190:
```rust
// SAFETY: The for loop (0..=MAX_RETRIES) always executes at least once.
// Every Err branch sets last_error = Some(err). If we reach here,
// all attempts returned Err, so last_error is always Some.
result: Err(last_error.expect("unreachable: loop always sets last_error")),
```
This keeps the `.expect()` but with a clear SAFETY comment. This is acceptable because it is provably unreachable.
