Output: `src/router/complexity.rs` with `score_complexity()` function, unit tests, and re-export from router module.

    **3. Create `src/router/complexity.rs`:**

    The file implements `pub fn score_complexity(messages: &[Message], weights: &ComplexityWeightsConfig) -> f64` (per D-12).

    **Default-to-frontier guard (per D-10):**
    If `messages.is_empty()` or total text content across all messages is < 10 chars, return `1.0` immediately.

    **Text extraction:**
    Concatenate all message text using `msg.content.as_str()` for each message. This handles both `MessageContent::Text` and `MessageContent::Parts` variants (per SCORE-04).

    **5 signal functions, each returning 0.0-1.0:**

    a) `signal_context_length(total_text: &str) -> f64` (per D-01, D-02):
       - Approximate tokens: `total_text.len() / 4`
       - Sigmoid centered at 1000 tokens (4000 chars): `1.0 / (1.0 + (-0.003 * (tokens as f64 - 1000.0)).exp())`
       - Short text -> ~0.0, long text -> ~1.0

    b) `signal_code_blocks(total_text: &str) -> f64` (per D-03):
       - Use `LazyLock<Regex>` (per D-07) for pattern: ` ```[\s\S]*?``` ` (fenced code blocks with triple backticks)
       - Count matches. 0 -> 0.0, 1 -> 0.3, 2 -> 0.6, 3+ -> 1.0 (smooth step: `(count as f64 / 3.0).min(1.0)`)

    c) `signal_multi_file(total_text: &str) -> f64` (per D-04):
       - Use `LazyLock<Regex>` for file path patterns: `\b\w+(?:/\w+)+\.\w{1,5}\b` (matches `src/main.rs`, `tests/foo.py`, etc.)
       - Also match standalone extensions pattern: `\.\b(?:rs|py|ts|js|go|java|cpp|c|h|toml|yaml|yml|json|md|sql|sh)\b`
       - Count unique matches. 0 -> 0.0, 1 -> 0.3, 2 -> 0.5, 3+ -> 1.0 (`(count as f64 / 3.0).min(1.0)`)

    d) `signal_reasoning_keywords(total_text: &str, extra_keywords: &[String]) -> f64` (per D-05):
       - Default keyword set (case-insensitive matching): `["architect", "design", "tradeoff", "refactor", "why does", "compare", "across the codebase", "step by step", "analyze", "evaluate", "debug"]`
       - Also check each string in `extra_keywords`
       - Count how many distinct keywords appear in `total_text.to_lowercase()`
       - 0 -> 0.0, 1 -> 0.3, 2 -> 0.5, 3+ -> 1.0 (`(count as f64 / 3.0).min(1.0)`)

    e) `signal_conversation_depth(message_count: usize) -> f64` (per D-06):
       - Sigmoid: `1.0 / (1.0 + (-0.5 * (message_count as f64 - 6.0)).exp())`
       - 1 message -> ~0.08, 6 messages -> 0.5, 12+ -> ~0.95

    **Weighted average (per D-08, D-09):**
    ```rust
    let signals = [
        (signal_context_length(&text), weights.context_length),
        (signal_code_blocks(&text), weights.code_blocks),
        (signal_multi_file(&text), weights.multi_file),
        (signal_reasoning_keywords(&text, &weights.extra_keywords), weights.reasoning_keywords),
        (signal_conversation_depth(messages.len()), weights.conversation_depth),
    ];
    let (weighted_sum, weight_total) = signals.iter()
        .fold((0.0, 0.0), |(ws, wt), (signal, weight)| (ws + signal * weight, wt + weight));
    if weight_total == 0.0 { return 1.0; } // all weights zero -> frontier
    (weighted_sum / weight_total).clamp(0.0, 1.0)
    ```

| T-17-01 | D (Denial of Service) | signal_context_length | mitigate | Scorer operates on string slices with O(n) regex -- no unbounded allocation. LazyLock compiles regex once. Scoring is sub-millisecond even for large inputs. |
| T-17-02 | T (Tampering) | signal_reasoning_keywords | accept | Adversarial keyword stuffing could inflate scores, but this only routes to MORE capable (frontier) models -- no security impact, just cost impact. Acceptable for heuristic tier bucketing. |
| T-17-03 | D (Denial of Service) | regex patterns | mitigate | All regex patterns are compile-time constants via LazyLock, no user-supplied regex. Patterns are simple (no catastrophic backtracking). |
| T-17-04 | I (Information Disclosure) | score_complexity | accept | Score value is a float derived from message content -- does not leak message content itself. Score is only used internally (exposed in Phase 20 headers, not this phase). |
