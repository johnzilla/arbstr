2. Create `tests/vault_billing.rs` with the following tests. Use `axum::Router` as a mini HTTP server for the mock vault (bind to localhost:0, get the assigned port). Each test:
   - Starts a mock vault server (axum app on a random port) that records requests and returns configurable responses
   - Creates a test app with vault pointing at the mock
   - Sends a chat completion request via `tower::ServiceExt::oneshot`
   - Asserts on the response status and the recorded vault calls

   The mock vault is an axum app with these routes:
   - `POST /internal/reserve` -- validates X-Internal-Token header matches "test-internal-token", extracts agent_token/amount_msats/model from body, returns `{"reservation_id": "res-{uuid}"}` with 200. For 402 test: return 402. For 403 test: return 403.
   - `POST /internal/settle` -- records the settle call, returns `{"settled": true, "refunded_msats": 0}` with 200
   - `POST /internal/release` -- records the release call, returns `{"released": true}` with 200

   Use `Arc<Mutex<Vec<serde_json::Value>>>` to record vault calls so tests can assert on them.

   **Tests to implement:**

   a. `test_vault_reserve_requires_bearer_token` -- Send request without Authorization header when vault is configured. Expect 401 with OpenAI error format `{"error": {"message": "Authorization: Bearer <token> required", ...}}`.

   b. `test_vault_reserve_insufficient_balance` -- Mock vault returns 402 on reserve. Expect 402 response with `"type": "billing_error"`.

   c. `test_vault_reserve_policy_denied` -- Mock vault returns 403 on reserve. Expect 403 response with `"type": "billing_error"`.

   d. `test_vault_reserve_rate_limited` -- Mock vault returns 429 on reserve. Expect 429 response with `"type": "billing_error"`.

   e. `test_vault_reserve_uses_frontier_rates` -- Mock vault returns success on reserve, record the `amount_msats` from the reserve request body. With providers at rates local(1,5,0) and frontier(10,30,2), and default_reserve_tokens=4096: expected input estimate ~10 tokens (short test message), output=4096. Reserve should use rates (10,30,2): `10*10 + 4096*30 + 2000 = 100 + 122880 + 2000 = 124980 msats` (approximately -- the exact input token estimate depends on message length, but verify amount_msats uses frontier rates not local rates by checking it's > what local rates would produce).

   f. `test_free_proxy_mode_no_vault` -- Create app WITHOUT vault config (vault: None). Send request without Authorization header. Expect the request to proceed (will fail at provider level since fake URL, but should NOT return 401 -- verifying free proxy mode per D-09/BILL-08).

   g. `test_vault_auth_replaces_server_auth` -- Create app with vault config AND auth_token set. Send request with a bearer token that does NOT match auth_token but IS a valid vault agent token. The request should NOT be rejected by server auth middleware (because vault is configured, server auth is disabled per D-01/D-02/BILL-06).

   For the provider response in success cases: since the test providers use fake URLs, the provider call will fail. This is fine for testing vault reserve/error behavior. The release path will be triggered by provider failure, which also exercises BILL-04.

| T-21-03 | Spoofing | agent_token in reserve | mitigate | Vault validates agent_token server-side; proxy only forwards it. Tests verify 401 when token missing. |

| T-21-05 | Denial of Service | vault rate limiting | mitigate | 429 from vault propagated to client. Backpressure check prevents request accumulation. |
